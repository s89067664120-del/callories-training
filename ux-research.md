## Часть 1. Три сценария взаимодействия

---

### Сценарий 1 — Первое знакомство (онбординг)

#### Шаги пользователя

1. Находит бота через рекламу, поиск или рекомендацию → нажимает **Start**
2. Читает приветственное сообщение (что умеет бот, зачем он нужен)
3. Отвечает на вопросы по одному: рост → вес → возраст → цель → активность
4. Получает рассчитанную норму калорий (TDEE + целевой дефицит)
5. Видит предложение сразу добавить первую еду или посмотреть дневник
6. Делает первое целевое действие — фотографирует еду или жмёт «Посмотреть дневник»

#### Что удобно (лучшие практики)

- **Вопросы по одному** — пользователь не видит «анкету из 10 пунктов» сразу, не пугается объёма
- **Inline-кнопки для выбора цели** («Похудеть / Поддержание / Набор»): не нужно ничего печатать, нет ошибок ввода
- **Немедленный результат после онбординга**: «Ваша норма — 1650 ккал в день. Давайте запишем первый приём пищи?» — пользователь сразу видит ценность, не уходит думать
- **Один экран — одна мысль**: каждое сообщение бота содержит ровно один вопрос или один вывод

#### Что раздражает (типичные проблемы)

- **Слишком длинный онбординг** — 5+ обязательных шагов перед первым полезным действием. Исследования показывают: каждый дополнительный шаг снижает конверсию онбординга на 10–20%. Пользователи r/loseit жалуются: «Я хотел просто сфотографировать еду, а меня час заставляли вводить данные»
- **Свободный ввод числовых данных без подсказок** — пользователь пишет «180» (рост), бот не понимает единицы (см? дюймы?), переспрашивает → раздражение
- **Нет возможности вернуться и исправить** — вписал неправильный вес, и теперь не найти, как его поменять
- **Перегруженное приветственное сообщение** — «список из 15 функций бота» в первом сообщении. Пользователь ещё не доверяет боту и не читает это
- **Отсутствие прогресс-индикатора** — непонятно, сколько ещё вопросов впереди. «Шаг 2 из 5» снижает тревогу

#### Типичные ошибки разработчиков

| Ошибка | Почему возникает | Как исправить |
|---|---|---|
| Все вопросы в одном сообщении | Разработчик хочет «сразу получить все данные» | Разбить на последовательные сообщения с ожиданием ответа |
| Нет кнопок для выбора активности | Лень делать UI | Inline-кнопки: «Сижу за компьютером / Хожу / Тренируюсь» |
| Онбординг без возможности пропустить | «Нам нужны данные для расчёта» | Разрешить пропустить → использовать дефолтные значения |
| После онбординга — просто «Готово!» | Нет понимания воронки | Сразу направить к первому действию (CTA: «Сфотографируйте первую еду») |
| Запрос email или регистрация | Скопировано из web-UX | В Telegram ID пользователя уже есть, email не нужен |

### Сценарий 2 — Быстрое добавление еды (ежедневное использование)

#### Шаги пользователя

1. Открывает чат с ботом (может пройти несколько часов с последнего визита)
2. Фотографирует тарелку и отправляет фото
3. Видит сообщение «Анализирую фото...» (индикатор работы)
4. Получает ответ: что распознано, КБЖУ, короткий совет
5. Нажимает «Добавить» или «Изменить» (если бот ошибся)
6. Видит обновлённый счётчик дня: «Съедено 840 / 1650 ккал»

**Целевое время сценария: под 30 секунд от отправки фото до закрытия чата.**

#### Что удобно (лучшие практики)

- **Немедленный feedback «Анализирую...»** — без него пользователь думает, что фото не дошло, и отправляет повторно (двойная запись!)
- **Три кнопки после распознавания: Добавить / Изменить / Не добавлять** — не нужно ничего печатать в штатном сценарии. Это соответствует текущей спецификации CalBot; альтернативный подход — показывать top-3 варианта распознавания сразу (без шага «Изменить») — снижает количество шагов при ошибке, но усложняет UI. Стоит проверить на реальных пользователях
- **Показ итога дня прямо после добавления** — «Осталось 810 ккал» закрывает петлю: пользователь видит прогресс здесь и сейчас
- **Распознавание нескольких блюд на фото** — если на тарелке суп и хлеб, бот перечисляет оба; пользователь видит, что бот «умный»
- **Короткий совет без морализаторства** — «Чтобы снизить калорийность, попробуй без сметаны» вместо «Это слишком жирное блюдо»

#### Что раздражает (типичные проблемы)

- **Бот не отвечает 15+ секунд без индикатора** — пользователь в панике, пишет «алло?», уходит
- **При ошибке распознавания предлагает написать название текстом** — пользователь не помнит, как называется блюдо по-английски, не знает формат ввода, пишет «куриный суп» → бот не парсит → ещё ошибка
- **Кнопки «протухают» после перезапуска бота** — пользователь нажимает «Добавить» через 10 минут, бот отвечает ошибкой (классическая проблема Telegram inline keyboards без state management)
- **Ответ бота слишком длинный** — абзац текста с историей блюда, советами по приготовлению и тремя ссылками. Пользователь хочет просто нажать «Добавить»
- **Нет быстрого добавления без фото** — «Я выпил кофе с молоком» → нельзя добавить без фото → пользователь не логирует → статистика рушится
- **Бот не обрабатывает нефудовые фото** — пользователь случайно отправляет скриншот или мем (особенно при переключении между чатами). Если бот пытается «распознать еду» на скриншоте или молча игнорирует → путаница. Нужен явный ответ: «Не вижу еды на фото. Попробуй сделать снимок покрупнее» с кнопкой «Ввести вручную»

#### Типичные ошибки разработчиков

| Ошибка | Почему возникает | Как исправить |
|---|---|---|
| Нет промежуточного «Анализирую...» | Разработчик не думает о latency | `bot.send_message("Анализирую фото...")` сразу при получении фото |
| Коррекция только через свободный текст | Сложно реализовать гибкий UI | Top-3 варианта распознавания как inline-кнопки + «Другое» |
| Кнопки не сохраняют контекст | Нет session/state storage | Хранить callback_data с ID записи в БД, не полагаться на глобальный стейт |
| Формат ответа — эссе | «Хочу дать максимум информации» | Жёсткий шаблон: блюдо + КБЖУ одной строкой + 3 кнопки |
| Нет текстового добавления | «Мы про фото» | Команда `/add` с текстовым поиском как fallback |

### Сценарий 3 — Просмотр прогресса и мотивация

#### Шаги пользователя

**Активный сценарий (пользователь инициирует):**
1. Открывает чат → нажимает «Прогресс» или пишет `/stats`
2. Видит сводку за текущую неделю: дней в норме, средние калории, динамику
3. Получает мотивационный комментарий от бота

**Пассивный сценарий (бот инициирует — вечерний отчёт):**
1. Вечером приходит сообщение от бота с итогом дня
2. Пользователь читает оценку, при желании — нажимает «Рекомендации на завтра»
3. Закрывает чат (или отвечает, если есть вопрос)

#### Что удобно (лучшие практики)

- **Вечерний пуш-отчёт, а не ожидание запроса** — пользователи забывают проверять прогресс сами; напоминание от бота в 20:00–21:00 даёт retention-буст по данным аналогичных продуктов
- **Конкретные числа + эмоциональная оценка** — «5 из 7 дней в норме — отличная неделя!» работает лучше, чем просто «Ваши калории за неделю: таблица»
- **Серии (streak)** — «Это твой 4-й день подряд в норме» создаёт эффект commitment: не хочется ломать серию
- **После «срыва» — поддержка, не осуждение** — «Вчера был перебор. Сегодня новый день — давай начнём заново» удерживает пользователей; «Ты превысил норму на 800 ккал» — отталкивает
- **Прогресс-бар текстом** — `[████████░░] 80% нормы` читается мгновенно в любом интерфейсе Telegram

#### Что раздражает (типичные проблемы)

- **Графики как изображения без объяснений** — бот присылает PNG-график без подписей. Пользователь не понимает, что он видит
- **Слишком частые пуши** — уведомления утром, в обед и вечером → пользователь отключает уведомления или блокирует бота
- **Статистика только «с начала использования»** — если пользователь начал 3 месяца назад и делал перерывы, итоговые числа выглядят плохо. Нужна фильтрация: за неделю, за месяц
- **Мотивация только при успехе** — бот молчит, когда плохо, и «поздравляет», когда хорошо. В реальной жизни нужна поддержка именно в провальные дни
- **Нет возможности скорректировать вес** — пользователь похудел на 3 кг, но не может это зафиксировать → норма калорий устарела → доверие к боту падает

#### Типичные ошибки разработчиков

| Ошибка | Почему возникает | Как исправить |
|---|---|---|
| Только pull-статистика (по запросу) | «Пользователь сам спросит» | Вечерний scheduled message через `apscheduler` |
| Push каждый час | «Больше engagement» | Максимум 1 пуш в день, желательно вечером |
| Таблица калорий текстом | Так проще реализовать | Текстовый прогресс-бар + 2–3 ключевых числа |
| Мотивация — одна заготовленная фраза | Лень писать варианты | Минимум 3 варианта сообщений под разные ситуации: успех / норма / перебор |
| Нет обновления профиля | Не предусмотрено в архитектуре | Команда `/profile` → редактирование веса, цели, нормы |

## Часть 2. UX-принципы для удержания пользователей

### 1. Правило 10 секунд на ежедневное действие

Основное действие (добавить еду) должно занимать не более 10–15 секунд: открыл → сфотографировал → нажал «Добавить». Каждая лишняя секунда или лишний шаг снижает вероятность, что пользователь будет возвращаться каждый день. Friction — главный враг habit-forming продуктов.

**Проверочный вопрос:** Сколько тапов нужно от открытия чата до записи еды? Должно быть ≤ 3.

---

### 2. Экономия сообщений (message economy)

Telegram — это мессенджер, а не приложение. Пользователь воспринимает каждое сообщение бота как «уведомление». Длинные, частые или бессодержательные сообщения вызывают усталость. Правило: одно сообщение бота = одна задача для пользователя.

Важно разграничивать два типа сообщений:
- **Пустые подтверждения** («Ок, понял!», «Отлично!» перед следующим сообщением) — балласт, удваивают количество сообщений, убирать
- **Обновления состояния** («Съедено 840 / 1650 ккал — осталось 810») — несут новую информацию и закрывают петлю обратной связи, оставлять. Именно такой подход заложен в CalBot: после добавления блюда бот сразу показывает итог дня — это правильно

**Проверочный вопрос:** Это сообщение меняет что-то в «картине мира» пользователя или просто сопровождает следующее сообщение? Если только сопровождает — убрать.

---

### 3. Немедленная обратная связь на любое действие

Пользователь должен получать ответ в течение 1–2 секунд после любого действия — или хотя бы подтверждение, что запрос принят («Анализирую...», «Загружаю...»). Без feedback-петли пользователь не понимает, сработало ли его действие, повторяет его, получает дубли или теряет доверие к боту.

**Критично для:** отправки фото (Vision API имеет latency 3–8 секунд), смены настроек, любой операции дольше 1 секунды.

---

### 4. Прощение ошибок (error forgiveness)

Бот для питания — это продукт, которым пользуются каждый день, часто второпях. Ошибки неизбежны: отправил не то фото, нажал не ту кнопку, добавил блюдо дважды. Система должна делать откат и исправление тривиальными: кнопка «Отменить последнее», возможность удалить запись, простое редактирование профиля. Отсутствие этих функций — одна из главных причин, по которым пользователи бросают трекеры.

**Минимальный набор:** удалить последнюю запись, исправить вес в профиле, пропустить вопрос онбординга.

---

### 5. Поддержка, а не аудит

Тональность бота критически влияет на retention. Calorie tracking — эмоционально нагруженная тема. Пользователи, которые едят «плохо», и так это знают; им не нужно ещё одно напоминание об этом от бота. Боты, которые осуждают («Вы превысили норму!»), теряют пользователей после первого срыва. Боты, которые поддерживают («Сегодня был сложный день. Завтра — новая страница»), удерживают пользователей через плохие дни — а именно в этих точках и происходит отток.

**Практическое правило:** никогда не используй слова «превышение», «нарушение», «слишком много». Используй «чуть больше нормы», «завтра скорректируем», «это нормально».

---

### 6. Прогрессивное раскрытие функций

Новый пользователь не должен видеть все возможности бота в первый день. Онбординг → добавление еды → итог дня — это достаточно для первого дня. Функции статистики, рецептов, тренировок, геймификации должны появляться постепенно: на 3-й день показать прогресс, на 7-й — ачивку за неделю, на 14-й — предложить обновить цель.

**Антипаттерн:** стартовое меню с кнопками «Дневник / Рецепты / Тренировки / Статистика / Профиль / Помощь» в день первого визита.

---

### 7. Пуш-сообщения как механизм возврата, а не раздражитель

Scheduled messages — единственный инструмент, которым бот может инициировать контакт. Это мощный retention-инструмент при правильном использовании и главная причина блокировки бота при злоупотреблении. Оптимум: 1 вечерний итог дня (20:00–21:00 по часовому поясу пользователя) + ситуативные сообщения (серия прервана, новая ачивка). Утренние «напоминалки» работают хуже вечерних отчётов — пользователь ещё не успел ничего съесть.

**Минимальный контроль пользователя:** возможность отключить пуши или изменить время через `/settings`.

## Часть 3. Специфика ограничений Telegram Bot API

Эти ограничения напрямую определяют UX — их нельзя обойти дизайном.

| Ограничение | Влияние на UX | Рекомендация |
|---|---|---|
| Бот не может инициировать диалог с новым пользователем | Онбординг начинается только после `/start` | Убедиться, что `/start` — всегда точка входа; не ломать при повторном нажатии |
| Inline-кнопки становятся неактивными при рестарте бота или истечении сессии | Пользователь нажимает «Добавить» через 20 минут — ошибка | Хранить контекст в БД по `callback_data`, обрабатывать устаревшие callback с понятным сообщением |
| Нет фоновой синхронизации — пользователь видит только то, что бот прислал | Нет «живого» обновления счётчика калорий | Всегда обновлять счётчик в следующем сообщении после добавления еды |
| Максимальный размер фото — 10 МБ, Vision API имеет latency | Пользователь ждёт ответа | Немедленный «Анализирую...» + обработка таймаута |
| История чата — единственный «экран» | Длинная история = потерянный контекст | Дублировать ключевые данные (норма, остаток) в каждом релевантном сообщении |

## Ключевые инсайты для CalBot

Проект в `project.md` уже правильно учитывает большинство критичных принципов — вопросы по одному, только кнопки, без ручного ввода. Три зоны риска, которые стоит проработать отдельно:

1. **Сценарий ошибки распознавания** — что происходит, если бот не распознал блюдо или распознал неверно? Нужны конкретные кнопки («Попробовать ещё раз / Ввести название / Пропустить»), а не просьба написать текстом.

2. **Тайм-менеджмент пушей** — вечерний отчёт нужно настраивать по часовому поясу пользователя; без этого сообщение «Итог дня» в 3 ночи — прямой путь к блокировке.

3. **Повторный онбординг** — пользователь, который вернулся через месяц перерыва, не должен снова проходить все 5 шагов. Нужна проверка: «Вы уже у нас! Продолжить или сбросить данные?»
