## 1. Распознавание еды по фото (AI Vision API)

### Варианты

| Критерий | Claude Vision (claude-opus-4 / sonnet) | GPT-4o Vision (OpenAI) | Logmeal Food AI API |
|---|---|---|---|
| **Тип** | General-purpose LLM с vision | General-purpose LLM с vision | Специализированный food AI |
| **Точность распознавания еды** | Высокая. Хорошо понимает контекст, описывает состав и способ приготовления | Высокая. Сравнимо с Claude, сильнее в западной кухне | Высокая для food-задач. Обучен специально на блюдах |
| **Нутриенты** | Считает сам по visual estimate — без привязки к USDA базе | Считает сам по visual estimate | Возвращает данные из food DB (более точно) |
| **Гибкость промптов** | Очень высокая: структурированный JSON, тон ответа, советы | Очень высокая | Ограничена — API отдаёт фиксированный формат |
| **Поддержка русской кухни** | Хорошая — знает борщ, пельмени, оливье | Хорошая | Слабее — база данных ориентирована на западную еду |
| **Цена за 1 запрос (фото ~300KB)** | ~$0.003–0.012 (Sonnet 3.5/4) | ~$0.005–0.015 (gpt-4o) | ~$0.02–0.05 (зависит от плана) |
| **Бесплатный тариф** | Нет. $5 кредит при регистрации (разовый) | Нет. $5 кредит при регистрации | 50–100 бесплатных запросов/мес (free tier) |
| **Задержка ответа** | 2–5 секунд | 2–6 секунд | 1–3 секунды |
| **Плюсы** | Отлично понимает контекст; легко дать любой тон и формат; интеграция с Claude для коуча сразу в одном API | Большая экосистема; хорошо задокументирован; много примеров на GitHub | Специализирован; возвращает нутриенты из базы; меньше нужно prompt engineering |
| **Минусы** | Visual estimate нутриентов — приблизительный; нет базы USDA | Visual estimate нутриентов — приблизительный | Слабая поддержка СНГ-кухни; ограниченная кастомизация ответов; менее гибкий |
| **Документация** | docs.anthropic.com | platform.openai.com/docs | docs.logmeal.com |

### Гибридная схема (рекомендуется для точности)

```
Фото → Claude/GPT-4o Vision → название блюда + вес (estimate)
                                      ↓
                           Nutritionix API / USDA FoodData Central
                                      ↓
                           Точные нутриенты из базы данных
```

**Nutritionix API** — крупнейшая база нутриентов:
- Natural Language endpoint: GET `/natural/nutrients` (текстовый запрос)
- Бесплатно: 500 запросов/день для разработчиков
- Платно: от $50/мес
- Хорошо покрывает ресторанные блюда и брендовые продукты

**USDA FoodData Central** — полностью бесплатный, без лимитов, но только для базовых продуктов (не блюда).

### Вывод по категории

Для MVP с минимальным бюджетом и русской аудиторией:
**Рекомендация: Claude Vision (claude-sonnet-3.5 или 3.7)** — он уже используется в проекте (см. project.md), хорошо знает СНГ-кухню, возвращает текст в любом формате, и тот же API можно использовать для AI-коуча. Visual estimate нутриентов приемлем для MVP (проект явно указывает «без точного подсчёта до грамма»). При росте — добавить Nutritionix для уточнения.

## 2. База данных

### Варианты

| Критерий | SQLite | PostgreSQL | MongoDB |
|---|---|---|---|
| **Тип** | Встроенная файловая СУБД | Реляционная СУБД | Документная NoSQL |
| **Схема данных** | Жёсткая (SQL) | Жёсткая (SQL) | Гибкая (JSON-документы) |
| **Установка** | Не нужна — входит в Python stdlib | Нужен сервер | Нужен сервер |
| **Хостинг** | Файл рядом с кодом | Managed: Railway, Supabase, Neon | Managed: MongoDB Atlas |
| **Бесплатный тариф** | Всегда бесплатна | Railway: $5 кредит; Supabase: 500MB; Neon: 0.5GB | Atlas: 512MB (M0) |
| **Производительность при < 10K пользователей** | Достаточна | Отлично | Отлично |
| **Производительность при > 100K пользователей** | Проблемы с конкурентными записями | Отлично | Отлично |
| **Запросы к данным** | SQL — агрегации, JOIN | SQL — мощные аналитические запросы | Aggregation pipeline — менее привычен |
| **Миграции схемы** | Alembic (неудобно) | Alembic — стандарт | Нет миграций (плюс и минус) |
| **ORM** | SQLAlchemy / Peewee | SQLAlchemy / Tortoise ORM | Motor (async), Beanie |
| **Подходящая структура данных** | Слабо (еда как JSON в поле — костыль) | Хорошо (отдельные таблицы) | Хорошо (документ с вложенным списком еды) |
| **Плюсы** | Нулевая настройка; легко бэкапить (один файл); идеально для прототипа | Надёжность; ACID; мощные запросы; стандарт для Python-проектов; хороший managed-хостинг | Гибкость схемы; удобно хранить список блюд за день как массив |
| **Минусы** | Нет конкурентных записей (один writer); нельзя масштабировать горизонтально; не подходит для продакшна с несколькими инстансами | Нужен отдельный сервис; чуть сложнее настройка | Менее привычен Python-разработчикам; нет транзакций уровня SQL; Beanie/Motor — меньше туториалов |
| **Рекомендуемый ORM/клиент** | SQLAlchemy + SQLite | SQLAlchemy async + asyncpg | Beanie (async ODM) |

### Структура данных для калорийного бота

```
users           → user_id, tg_id, weight, height, age, goal, tdee, created_at
food_entries    → id, user_id, date, meal_name, calories, protein, fat, carbs, photo_url, created_at
daily_summaries → id, user_id, date, total_calories, goal_calories, status
```

Эта реляционная структура одинаково хорошо ложится и на SQLite, и на PostgreSQL.

### Вывод по категории

- **Прототип / первые 100 пользователей**: SQLite — нет ничего проще. Файл можно хранить на Railway/VPS рядом с кодом.
- **MVP с ростом**: PostgreSQL на Supabase (бесплатно 500MB) или Neon (serverless PostgreSQL, бесплатно 0.5GB) — без смены кода ORM при переходе от SQLite.
- **MongoDB** — избыточен для этой задачи. Реляционная структура дневника питания лучше ложится на SQL.

**Рекомендация: начать с SQLite → мигрировать на PostgreSQL (Supabase/Neon) при первых признаках роста.** Миграция через Alembic занимает 1–2 часа при правильно написанном ORM-коде.

## 3. Хостинг бота

### Варианты

| Критерий | Railway | Render | Fly.io | VPS Hetzner / DigitalOcean |
|---|---|---|---|---|
| **Тип** | PaaS (Git deploy) | PaaS (Git deploy) | PaaS (контейнеры) | IaaS (голый сервер) |
| **Бесплатный тариф** | $5 кредит при регистрации (одноразовый), затем ~$5/мес | Free tier: 750 часов/мес (1 сервис), спит после 15 мин неактивности | Free tier: 3 shared VM (по 256MB RAM) | Нет |
| **Минимальная цена** | ~$5/мес (Starter) | $7/мес (без sleep) | $1.94/мес (shared-cpu-1x, 256MB) | €3.79/мес (CX11, 2GB RAM, Hetzner) |
| **RAM для бота** | 512MB–1GB на Starter | 512MB на платном | 256MB–1GB | Зависит от плана |
| **Деплой** | git push → автодеплой | git push → автодеплой | flyctl deploy (CLI) | SSH + ручной деплой или GitHub Actions |
| **PostgreSQL** | Встроенный managed PostgreSQL | Встроенный managed PostgreSQL | Не встроен (внешний) | Установить самому или внешний |
| **Сложность настройки** | Минимальная | Минимальная | Средняя (нужен Dockerfile) | Высокая (nginx, systemd, файрволл) |
| **Логи и мониторинг** | Встроены | Встроены | Встроены | Только то, что настроишь сам |
| **Масштабирование** | Простое (слайдер) | Простое | Простое | Ручное (новый сервер) |
| **Плюсы** | Самый простой старт для Python; встроенная БД; хороший DX | Простой деплой; хороший free tier для тестов | Дешевле Railway на продакшне; global edge | Полный контроль; дешевле всего при масштабе; нет vendor lock-in |
| **Минусы** | После $5 кредита — платно; нет постоянного free tier | Free tier усыпляет сервис (бот перестаёт отвечать) | Нужен Docker; менее удобен для быстрого старта | Много ручной работы; нужно знать Linux; нет автодеплоя из коробки |
| **Подходит для** | MVP, тест идеи | Только для тестов (из-за sleep) | Продакшн с экономией | Продакшн при > 500 пользователей |

### Важно про Render Free Tier

Render усыпляет бесплатные сервисы через 15 минут неактивности. Телеграм-бот на polling перестаёт получать сообщения. **Не подходит для production polling-бота.** Webhook-режим частично решает проблему, но первое сообщение после сна придёт с задержкой 30–60 секунд.

### Вывод по категории

- **Для старта и MVP**: Railway — деплой за 5 минут, встроенная PostgreSQL, автодеплой из GitHub. Первые несколько недель покрывает $5 кредит.
- **Для экономии на продакшне**: Hetzner CX22 (€4.35/мес, 4GB RAM, 2 vCPU) — полный контроль, самый дешёвый вариант для стабильной работы.
- **Fly.io** — хороший компромисс, но порог входа выше из-за Docker.

**Рекомендация: Railway для MVP → Hetzner VPS при первых 200+ активных пользователях.**

## 4. Уведомления и планировщики задач

Бот по ТЗ отправляет вечерний итог дня и периодически пишет сам (AI-коуч).

### Варианты

| Критерий | APScheduler | Celery + Redis | python-telegram-bot JobQueue |
|---|---|---|---|
| **Тип** | Библиотека планировщика (в процессе бота) | Очередь задач (отдельный процесс) | Встроенный планировщик PTB |
| **Установка** | `pip install apscheduler` | `pip install celery redis` + Redis сервер | Встроен в python-telegram-bot |
| **Сложность** | Низкая | Высокая | Минимальная |
| **Персонализированные напоминания** | Да (отдельная задача на user_id) | Да (задачи в очереди) | Да (через `job_queue.run_daily`) |
| **Надёжность при перезапуске** | Задачи теряются (если не persist-хранилище) | Задачи в Redis — не теряются | Задачи теряются |
| **Масштабирование** | Только вертикально (один процесс) | Горизонтально (несколько воркеров) | Только вертикально |
| **Нужен Redis** | Нет | Да | Нет |
| **Плюсы** | Просто; не нужен Redis; работает в одном процессе | Надёжно; масштабируется; retry при ошибках | Нулевая настройка если уже используешь PTB |
| **Минусы** | Задачи в памяти — при крэше теряются; не масштабируется | Сложно для MVP; нужен отдельный сервис Redis | Нет retry; нет персистентности |
| **Подходит для** | MVP с несколькими сотнями пользователей | Продакшн с тысячами пользователей | Простые периодические задачи |

### Схема для MVP

```python
# APScheduler — ежедневный итог в 21:00 по часовому поясу пользователя
scheduler.add_job(send_daily_summary, 'cron', hour=21, minute=0, args=[user_id])

# Или через python-telegram-bot JobQueue
application.job_queue.run_daily(
    send_daily_summary,
    time=datetime.time(21, 0, tzinfo=user_tz),
    chat_id=user_chat_id
)
```

### Вывод по категории

**Рекомендация: APScheduler для MVP.** Прост в интеграции, не требует Redis, хорошо работает при сотнях пользователей. Для персистентности задач добавить `APScheduler` с `SQLAlchemyJobStore` (хранит задачи в той же БД). При росте до тысяч пользователей — мигрировать на Celery.

## 5. Фреймворк для Telegram-бота

### Варианты

| Критерий | python-telegram-bot (PTB) | aiogram | pyTelegramBotAPI (telebot) |
|---|---|---|---|
| **GitHub Stars** | ~26K | ~15K | ~8K |
| **Версия (2025)** | 21.x | 3.x | 4.x |
| **Async** | Да (с v20+, asyncio) | Да (полностью async) | Нет (по умолчанию). Async через `asyncio_helper` |
| **Архитектура** | Классы Handler + Application | Router + Middleware + FSM | Декораторы |
| **FSM (конечные автоматы)** | ConversationHandler (встроен) | FSMContext (мощный, встроен) | Нет встроенного (нужна библиотека) |
| **Middleware** | Да | Да (более гибкие) | Ограниченно |
| **Документация** | Отличная. docs.python-telegram-bot.org | Хорошая. docs.aiogram.dev | Базовая |
| **Примеры на GitHub** | Очень много | Много | Много (но устаревших) |
| **Webhook + polling** | Оба режима | Оба режима | Оба режима |
| **Скорость разработки** | Средняя (больше шаблонного кода) | Быстрая (роутеры, меньше кода) | Быстрая (декораторы) |
| **Плюсы** | Самая зрелая; огромное сообщество; отличная документация; ConversationHandler идеален для онбординга | Современная архитектура; отличный FSM; популярна в рунете; активное развитие | Самый простой старт; хорош для скриптов |
| **Минусы** | Чуть более verbose; ConversationHandler сложен при большом числе состояний | Меньше англоязычных туториалов; миграции между версиями ломают API | Нет async из коробки; слабый FSM; меньше поддержки |
| **Подходит для** | MVP с онбордингом, диалогами, кнопками | Продакшн, высокая нагрузка, сложная логика | Простые боты, прототипы |

### Пример ConversationHandler (python-telegram-bot) для онбординга

```python
conv_handler = ConversationHandler(
    entry_points=[CommandHandler("start", start)],
    states={
        ASK_WEIGHT: [MessageHandler(filters.TEXT, ask_weight)],
        ASK_HEIGHT: [MessageHandler(filters.TEXT, ask_height)],
        ASK_AGE:    [MessageHandler(filters.TEXT, ask_age)],
        ASK_GOAL:   [CallbackQueryHandler(ask_goal)],
    },
    fallbacks=[CommandHandler("cancel", cancel)],
)
```

### Пример FSMContext (aiogram) для онбординга

```python
class OnboardingForm(StatesGroup):
    weight = State()
    height = State()
    age = State()
    goal = State()

@router.message(OnboardingForm.weight)
async def process_weight(message: Message, state: FSMContext):
    await state.update_data(weight=message.text)
    await state.set_state(OnboardingForm.height)
    await message.answer("Теперь введи рост (в см):")
```

### Вывод по категории

**Рекомендация: python-telegram-bot v21.** Причины:
- Лучшая документация на английском (проще найти ответы)
- ConversationHandler идеально подходит для пошагового онбординга из ТЗ
- Встроенный JobQueue для вечерних уведомлений
- Самое большое сообщество → быстрее решать баги
- aiogram — отличная альтернатива, особенно если команда знает её

## Итог: Рекомендуемый стек для MVP

### Схема стека

```
Пользователь → Telegram
                  ↓
         python-telegram-bot v21
                  ↓
         Claude Vision API (claude-sonnet-3.5/3.7)
         (распознавание фото + AI-коуч)
                  ↓
         SQLite → PostgreSQL (Supabase/Neon при росте)
                  ↓
         APScheduler (вечерние отчёты)
                  ↓
         Railway (хостинг) → Hetzner VPS (при росте)
```

### Таблица финального выбора

| Компонент | Выбор для MVP | Альтернатива при росте |
|---|---|---|
| **Фреймворк бота** | python-telegram-bot v21 | aiogram 3.x |
| **Распознавание еды** | Claude Vision (claude-sonnet-3.5) | + Nutritionix для уточнения нутриентов |
| **База данных** | SQLite + SQLAlchemy async | PostgreSQL на Neon/Supabase |
| **Планировщик** | APScheduler 3.x + SQLAlchemyJobStore | Celery + Redis |
| **Хостинг** | Railway | Hetzner CX22 (€4.35/мес) |
| **ORM** | SQLAlchemy 2.x (async) | Без изменений |

### Примерный бюджет MVP в месяц

| Статья | Стоимость |
|---|---|
| Railway Starter | $5/мес |
| Claude API (claude-sonnet-3.5), 1000 запросов/мес × $0.005 | ~$5/мес |
| База данных (SQLite на Railway, включена) | $0 |
| Telegram Bot API | Бесплатно |
| **Итого** | **~$10/мес** |

При 100 активных пользователях и ~3 фото в день = ~9000 запросов/мес × $0.005 = ~$45/мес на API.
Основная статья расходов при росте — Claude Vision API. При высокой нагрузке рассмотреть кэширование или батчинг.

### Порядок реализации MVP

1. Настроить python-telegram-bot + ConversationHandler для онбординга
2. Подключить Claude Vision API — обработчик фото с JSON-ответом (блюдо, ккал, БЖУ)
3. SQLite: создать таблицы users, food_entries через SQLAlchemy
4. Кнопки «Добавить / Изменить / Не добавлять» через InlineKeyboardMarkup
5. Показ итогов дня после каждого добавления
6. APScheduler: вечерний отчёт в 21:00
7. Деплой на Railway через GitHub
